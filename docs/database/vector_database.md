# Overview

One of the biggest architectural improvements introduced in Phase 4 was the addition of vector database capabilities.

Traditional relational databases are designed to store structured information such as numbers, text, dates, and relationships. While they excel at transactional workloads, they are not designed to understand the semantic meaning of natural language.

To enable semantic memory retrieval, PostgreSQL was extended using the **pgvector** extension. This allows the database to store vector embeddings generated by a Machine Learning model and perform similarity searches directly inside SQL.

With the completion of **Phase 5**, the vector database now serves as the retrieval layer of the Retrieval-Augmented Generation (RAG) pipeline.

Instead of returning search results directly to the client, retrieved memories are supplied to a Large Language Model (Gemini) as contextual information, allowing the assistant to generate personalized and memory-grounded responses.

The project therefore combines the reliability of a relational database with the capabilities of a vector database while keeping the entire AI retrieval pipeline inside PostgreSQL.

---

# Why a Vector Database?

Traditional databases compare text using exact values.

Example:

Stored memory

```
I have an interview tomorrow at 10 AM.
```

User query

```
What important work do I have tomorrow?
```

A traditional SQL query searches for matching keywords.

```sql
SELECT *
FROM memories
WHERE content LIKE '%interview%';
```

Since the word **interview** is not present in the query, no result is returned.

A vector database compares meanings instead of words.

Both sentences are converted into numerical vectors.

```
Memory

↓

Embedding

↓

Vector Space
```

The database retrieves vectors that are closest to the query vector.

---

# What is an Embedding?

An embedding is a dense numerical representation of text.

Example

```
"I have an interview tomorrow."
```

↓

```
[-0.080,
0.089,
0.039,
...
384 values]
```

Each value represents a learned feature extracted by the Machine Learning model.

Although humans cannot interpret these numbers directly, vectors with similar meanings occupy nearby positions in high-dimensional space.

---

# Why pgvector?

There are many dedicated vector databases available today, including:

- Pinecone
- Weaviate
- Milvus
- Qdrant
- ChromaDB

However, this project uses **pgvector** because it offers several advantages:

- Native PostgreSQL extension
- Open source
- Easy SQL integration
- ACID transactions
- No additional infrastructure
- Excellent for medium-scale semantic search
- Supports vector indexing
- Seamless integration with SQLAlchemy

This allows PostgreSQL to function as both the transactional database and the vector database.

# Why 384 Dimensions?

The selected embedding model

```
all-MiniLM-L6-v2
```

produces vectors with **384 dimensions**.

The database column must exactly match the model output.

```
VECTOR(384)
```

Changing the embedding model in the future may require a different vector dimension.

---

# SQLAlchemy Integration

The Python pgvector package integrates directly with SQLAlchemy.

Example

```python
from pgvector.sqlalchemy import Vector

embedding = mapped_column(
    Vector(384),
    nullable=True,
)
```

This allows SQLAlchemy models to work seamlessly with PostgreSQL vector columns.

A dedicated Alembic migration was created to add vector support to the database.

Migration Name

The migration added the following column:


Migration applied using:

```bash

```

The existing database schema was upgraded without affecting previously stored user data.

When the user asks a question, the application performs semantic search instead of keyword matching.

Workflow

```
User Question

↓

Generate Query Embedding

↓

Query Vector

↓

Cosine Similarity Search

↓

Nearest Memories
```

Unlike keyword search, semantic search compares meanings rather than exact words.

---

# Cosine Similarity

Cosine similarity measures how closely two vectors point in the same direction.

Identical vectors

```
Similarity = 1.0
```

Completely unrelated vectors

```
Similarity ≈ 0
```

The project uses pgvector's built-in cosine distance implementation.

Example

```python
Memory.embedding.cosine_distance(query_embedding)
```

The returned distance is converted into a similarity score before being returned by the API.

# Role in Retrieval-Augmented Generation (RAG)

Beginning with Phase 5 and enhanced further in Phase 6, the vector database acts as the long-term knowledge retrieval layer of the Retrieval-Augmented Generation (RAG) architecture.

Instead of allowing the Large Language Model to answer only from its pretrained knowledge, the backend first retrieves semantically relevant memories from PostgreSQL.

Those memories are combined with recent conversation history and inserted into a structured prompt before the request is sent to Gemini.

This approach offers several advantages:

- Reduces hallucinations
- Produces personalized responses
- Grounds answers in user memories
- Improves factual consistency
- Keeps AI responses explainable

# Current Limitations

The current implementation performs an exact nearest-neighbor search.

Advantages

- Highest retrieval accuracy
- Simple implementation
- Easy debugging

Disadvantages

- Slower for extremely large datasets
- No approximate indexing yet
- Limited context selection strategy

For the current project size, this implementation is more than sufficient.

# Why PostgreSQL Instead of Pinecone?

This project intentionally keeps both structured data and vector embeddings inside a single PostgreSQL database.

Benefits include:

- Simpler system architecture
- Easier deployment
- ACID transactions
- Lower infrastructure cost
- Fewer external dependencies
- Better consistency between metadata and embeddings
- Seamless SQLAlchemy integration

Although dedicated vector databases such as Pinecone, Weaviate, Milvus, or Qdrant offer advanced scalability features, PostgreSQL with pgvector is an excellent choice for small- to medium-scale AI systems and portfolio projects.
