# Overview

One of the biggest architectural improvements introduced in Phase 4 is the addition of vector database capabilities.

Traditional relational databases are designed to store structured information such as numbers, text, dates, and relationships. While they excel at transactional workloads, they are not designed to understand the semantic meaning of natural language.

To enable semantic memory retrieval, PostgreSQL was extended using the **pgvector** extension. This allows the database to store vector embeddings generated by a Machine Learning model and perform similarity searches directly inside SQL.

As a result, the project combines the reliability of a relational database with the capabilities of a vector database, eliminating the need for an external vector storage system.

---

# Why a Vector Database?

Traditional databases compare text using exact values.

Example:

Stored memory:

```
I have an interview tomorrow at 10 AM.
```

User query:

```
What important work do I have tomorrow?
```

A traditional SQL query searches for matching keywords.

```sql
SELECT *
FROM memories
WHERE content LIKE '%interview%';
```

Since the word **interview** is not present in the query, no result is returned.

A vector database compares meanings instead of words.

Both sentences are converted into numerical vectors.

```
Memory

↓

Embedding

↓

Vector Space
```

The database retrieves vectors that are closest to the query vector.

---

# What is an Embedding?

An embedding is a dense numerical representation of text.

Example:

```
"I have an interview tomorrow."
```

↓

```
[-0.080,
 0.089,
 0.039,
 ...
 384 values]
```

Each value represents a learned feature extracted by the Machine Learning model.

Although humans cannot interpret these numbers directly, vectors with similar meanings occupy nearby positions in high-dimensional space.

---

# Why pgvector?

There are many dedicated vector databases available today, including:

- Pinecone
- Weaviate
- Milvus
- Qdrant
- ChromaDB

However, this project uses **pgvector** because it offers several advantages:

- Native PostgreSQL extension
- Open source
- Easy SQL integration
- ACID transactions
- No additional infrastructure
- Excellent for medium-scale semantic search
- Supports vector indexing
- Seamless integration with SQLAlchemy

This makes PostgreSQL both the transactional database and the vector database.

# pgvector Installation

The extension was manually installed for PostgreSQL 17 on Windows.

After installation, it was enabled inside the project database.

```sql
CREATE EXTENSION vector;
```

Verification:

```sql
SELECT extname
FROM pg_extension;
```

Output:

```
vector
```


---

# Why 384 Dimensions?

The selected embedding model

```
all-MiniLM-L6-v2
```

produces vectors with **384 dimensions**.

The database column must exactly match the model output.

```
VECTOR(384)
```

Changing the embedding model in the future may require a different vector dimension.

---

# SQLAlchemy Integration

The Python pgvector package integrates directly with SQLAlchemy.

Example:

```python
from pgvector.sqlalchemy import Vector

embedding = mapped_column(
    Vector(384),
    nullable=True,
)
```

This allows SQLAlchemy models to work seamlessly with PostgreSQL vector columns.

# Alembic Migration


# Similarity Search

When the user performs semantic search:

```
User Question

↓

Embedding Generation

↓

Query Vector

↓

Cosine Similarity

↓

Nearest Memories
```

Unlike keyword search, this compares the meaning of sentences.

---

# Cosine Similarity

Cosine similarity measures how similar two vectors are.

Two identical vectors:

```
Similarity = 1.0
```

Unrelated vectors:

```
Similarity ≈ 0
```

The application uses cosine distance internally through pgvector.

Example:

```python
Memory.embedding.cosine_distance(query_embedding)
```

This produces the semantic ranking used by the search API.

---

# Query Execution

Current search pipeline:

```
Generate Query Embedding

↓

Calculate Cosine Distance

↓

Sort Results

↓

Return Top-K Memories
```

The closest vectors are returned first.

# Current Limitations

The current implementation performs an exact nearest-neighbor search.

Advantages:

- Highest accuracy
- Simple implementation

Disadvantages:

- Slower for extremely large datasets

For the current project size, this approach is more than sufficient.

# Why PostgreSQL Instead of Pinecone?

This project intentionally keeps both structured and vector data in a single database.

Benefits:

- Simpler architecture
- Easier deployment
- ACID transactions
- Lower operational cost
- Fewer external dependencies
- Better consistency between metadata and embeddings

Dedicated vector databases may be introduced in future versions if scalability requirements increase.
