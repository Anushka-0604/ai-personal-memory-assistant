# Overview

Phase 4 marks the transition of the AI Personal Memory & Decision Assistant from a traditional CRUD-based application to an AI-powered semantic memory system.

In the previous phases, the application was capable of storing, updating, deleting, and retrieving memories using conventional database queries. Although functional, this approach relied entirely on exact keyword matching and lacked the ability to understand the semantic meaning of stored information.

The primary objective of this phase was to integrate Machine Learning into the backend by introducing vector embeddings and semantic search. Every memory is now converted into a numerical vector representation using a pre-trained Sentence Transformer model. These embeddings are stored inside PostgreSQL using the pgvector extension, allowing the application to retrieve memories based on meaning instead of exact textual matches.

This phase lays the foundation for Retrieval-Augmented Generation (RAG), which will be implemented in the next phase.

The primary objectives of Phase 4 were:

# Why Semantic Search?

Traditional databases search using exact text matching.

For example:

User Query:

```
Interview tomorrow
```

SQL Query:

```sql
SELECT *
FROM memories
WHERE content LIKE '%interview%';
```

This approach only works if the stored memory contains the exact keyword.

Example:

Stored Memory:

```
I have an interview tomorrow at 10 AM.
```

Search Query:

```
What important work do I have tomorrow?
```

A traditional SQL search would fail because the word **interview** is not present in the user's query.

Semantic search solves this problem by comparing the meaning of sentences rather than their exact wording.

---

# Machine Learning Approach

Instead of storing only text, every memory is converted into a numerical representation called an **embedding**.

Example:

```
"I have an interview tomorrow."
```

↓

```
[-0.0803,
 0.0895,
 0.0395,
 ...
 384 values]
```

These vectors capture the semantic meaning of the sentence.

Memories with similar meanings produce vectors that are close together in vector space.

The migration included:

- Installing PostgreSQL 17
- Creating a new database
- Configuring PostgreSQL on port 5433
- Updating the backend configuration
- Re-running Alembic migrations

Database created:

```
ai_personal_memory
```

# Installing pgvector

PostgreSQL cannot store vector embeddings by default.

To support Machine Learning features, the pgvector extension was installed.

The extension provides:

- vector datatype
- cosine distance
- L2 distance
- inner product
- vector indexing

Extension installation:

```sql
CREATE EXTENSION vector;
```

After installation PostgreSQL became capable of storing dense embeddings generated by the Machine Learning model.

# Semantic Search Implementation

A new service function was implemented:

```
search_memories()
```

Instead of keyword matching, the algorithm performs vector similarity search.

Pipeline:

```
User Query

↓

Generate Query Embedding

↓

Compare with Stored Embeddings

↓

Cosine Similarity

↓

Return Most Relevant Memories
```

This enables intelligent retrieval even when the query does not contain the same keywords as the stored memory.

---

# Search API

A new endpoint was introduced.

```
POST /memories/search
```

Example request:

```json
{
    "query": "What important work do I have tomorrow?",
    "top_k": 5
}
```

Example response:

```json
[
    {
        "id": 1,
        "content": "I have an interview tomorrow at 10 AM.",
        "source": "chat",
        "similarity": 0.94
    }
]
```

This endpoint forms the retrieval layer required for Retrieval-Augmented Generation (RAG).
